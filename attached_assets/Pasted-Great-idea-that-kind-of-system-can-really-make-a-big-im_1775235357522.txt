IKODI Student Sponsorship Management System Requirements

1. User Authentication and Access Control

The system should allow users to log in only with system-generated usernames and passwords.

Users of the system include: Secretary, Admin, and Super Admin.

Access should be restricted based on the user role:

Secretary: Can perform only the standard operations defined for their role.

Admin: Has full control over system operations and can manage data according to access rights.

Super Admin: Has all admin privileges plus oversight of system activity logs and user management.




2. Dashboard Features
The dashboard should include the following functionalities:

Student Management: Add new students, record student information, and view student lists.

Sponsorship Management: Record sponsors, sponsorships, and payments (including academic reports).

Printing Options: Print individual or complete lists for students, sponsors, payments, and reports.

Term-Based Records: The system should only allow data entry and management for students within the same academic term to avoid mixing records across terms.



3. Student Records Management

There should be a section to upload and view student reports and records, ensuring that different types of records (academic, financial, personal) remain separated and organized.

Schools should be categorized into Primary Schools and Higher Learning Institutions.

For each student, the system should store:

Personal Information (Name, Date of Birth, Gender, etc.)

Contact Information (Phone number, Email)

Guardian Information (Name, Relationship, Contact Information)


The system should allow search functionality to quickly locate student or guardian records.

Admins should be able to export student and guardian information, including phone numbers and emails, into reports.



4. Printing and Branding

Any printed document or report from the system should include:

The IKODI  logo

Footer: “IKODI All rights reserved 2026.”




5. System Audit and History

Admin and Super Admin should have access to a complete history of system activity, including:

Who accessed the system

Access time

Actions performed




6. Comprehensive Student Records

All student records should cover complete academic history, from primary school through secondary school to higher learning institutions.



7. Communication Features

System users should be able to send and receive messages to both students and guardians through email and/or SMS using the contact information provided.



8. Data Update and Retrieval

Administrators should be able to update or modify records, with all changes tracked and retrievable according to user roles.
